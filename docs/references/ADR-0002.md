# ADR-0002: Claude Desktop【Code】タブの対応環境を Local に限定

- **Status**: Accepted (2026-06-29)
- **関連**: 基本設計書 §2-1・§3.1・§3.3・§11（結論の所在）／ §5-3（上流決定：localhost で1インスタンス共有）

## Context

kage の対象面 (c) は「Claude Desktop の【Code】タブ」である。設計初期、(c) は (b) Windows Terminal と同じ Windows 側 Claude Code 設定を共有する可能性が高いと仮説していた。この仮説が正しければ、実際に割れている軸は3面ではなく **2つの OS（WSL / Windows）** に縮み、(c) を独立した面として検出・配線する必要はなくなる。MVP の差別化も「クロス OS で1サーバを共有する配線の難しさ」に集約できる、という見通しだった。

この「(c) ≒ (b)」という前提のまま設計を進めるか、検証してから決めるかが分岐点だった。

## Decision

(c) Claude Desktop【Code】タブの対応を **Local 環境に限定**する。【Code】タブが起動時に選べる Local / Remote / SSH のうち、Remote / SSH は対象外とする。

> 本 ADR の射程は【Code】タブの**環境選択軸**に限る。CLAUDE.md 機構を持たない Desktop アプリ本体そのものの追加は別判断であり、コンテスト提出版の差別化（§3.2）として扱う。

## 棄却した代替案

### 案1：実機検証を経ず、(c) を (b) と同一の Windows 面とみなして進める

設計初期に実際に置いていた当初仮説そのもの。これを採れば (c) の独立検出も独立配線も不要で、設計は単純になる。**実際にこの前提の下で設計を進めていた**ため、これは後付けの藁人形ではなく、コミットしえた現実の分岐である。

- **棄却の根拠（実機検証 2026-06-24）**：【Code】タブは Local / Remote / SSH という独立した実行環境の選択軸を持つことが判明した。
    - **Local**：Windows のローカルファイルシステムを直接参照し、`~/.claude/CLAUDE.md` を読むことを実機確認。配線先は (b) に収束する。
    - **Remote / SSH**：Anthropic ホストのクラウド、または SSH 先の別マシンで動作し、ローカルファイルシステムを参照しない。
- 「(c) ≒ (b)」は **Local に限ってのみ成立**し、環境選択次第で別ファイルシステムへ分岐しうる。よって (c) は「(b) に畳める従属面」ではなく、**環境軸を持つ独立した面**として扱う必要がある。未検証のまま同一視する案は棄却。

### 案2：Remote / SSH 環境も配線対象に含める

Remote 環境で Desktop を使う利用者も自分のコンテキストを求めるため、「含める」は実在する要望ではある。

- **棄却の根拠（構造的不達）**：kage は HTTP（localhost）で1インスタンスを OS 間共有する設計（§5-3）。Remote / SSH は別マシン／クラウドで動作するため、localhost 到達経路が**構造的に届かない**。仮に MCP 設定を配線しても、ローカルの `~/.claude/` を参照しないため読まれない。原理的に不達であり、配線しても成功の定義を満たせないため棄却。

### 2つの棄却軸の関係

- 案1 は **認識論的**な棄却：「前提を検証せず進む」という_方法_を退け、実機検証を選んだ。
- 案2 は **構造的**な棄却：上流決定 §5-3（localhost 共有）の_帰結_として不達が確定する。
- 性質の違う2案を退けた結果として、対応範囲が Local の一点に確定する。片方だけでは「Local に限定」までは導けない——案1 のみなら Remote / SSH の扱いが宙に残り、案2 のみなら (c) を独立面として扱う必要性が立たない。

## Consequences

- §2-1 のゴール表で (c) を **Local 環境のみ**に修正した。§11 対応環境・§3.3 やらないこと、にも反映済み。
- Local 環境の配線先は (b) Windows Terminal と同じ Windows グローバル `~/.claude/CLAUDE.md` に収束する。したがって (c)-Local は**独立した配線ターゲットを増やさない**——面検出（§9-1）では (b) と同じ書き込み先に解決される。
- Remote / SSH を構造的不達として明示したことで、将来「なぜ Desktop を全環境で配線しないのか」という蒸し返しに対し、本 ADR が回答になる。
- **Arcana 昇格候補**：この決定は「初期仮説を実機検証で訂正し、当初仮説を消さずに併記して残す」という型の実例である。Arcana.md の「設計書＝意思決定の記録」原則（§8）を、_実機検証で覆った仮説の保存_という形へ具体化する昇格候補として残す。
