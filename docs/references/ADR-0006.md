# ADR-0006: (c) Desktop【Code】タブの connection 配線も (b) に収束（独立 `desktop` ターゲットを却下）

- **Status**: Accepted (2026-06-30)
- **関連**: 基本設計書 §8-5（結論の所在）／ §8-3・§9-1・§7-4／ ADR-0002（prompt 側 (c)→(b) 収束の延長）

## Context

ADR-0002 で、(c) Desktop【Code】タブ Local の **prompt（CLAUDE.md）** 配線は (b) Windows Terminal と同じ Windows グローバル `~/.claude/CLAUDE.md` に収束し、**独立した配線ターゲットを増やさない**ことが実機確認済みだった（ADR-0002 Consequences）。

ところが §8-5 の registry 構造は **connection（MCP 設定 JSON）** 側で `desktop` を独立ターゲットとし、`claude_desktop_config.json` へ配線する想定を残していた。§8-5 自身が「connection 側の各設定ファイルの正確な所在は面ごとに異なり…検出して実パスを台帳へ焼き付ける」と**残課題**として明記しており、prompt は (b) 収束・connection は独立、という非対称が宙に浮いていた。実装着手にあたり、この所在を実機で確定する必要があった。

## Decision

connection 配線も prompt と同じく **(b) に収束**させ、独立 `desktop` ターゲットを廃する。

- connection ターゲットは **`wsl` + `windows` の2系統**（prompt と同型）。
- 配線先ファイルは各面の **`$HOME/.claude.json` のトップレベル `mcpServers`** キー。形式は：
  ```json
  "mcpServers": { "kage": { "type": "http", "url": "http://127.0.0.1:<PORT>/mcp" } }
  ```
- これは §8-5 の「connection 所在は残課題」を**解消する確定**でもある（配線先は `settings.json` でも `claude_desktop_config.json` でもなく `.claude.json`）。

## 棄却した代替案

### 案：§8-5 当初想定どおり (c) を `claude_desktop_config.json` に独立配線する

- **棄却根拠1（実機）**：当該ファイルは `mcpServers` を持たない。実体は Desktop アプリ本体の preferences 設定（`coworkUserFilesPath` 等）であって、【Code】タブの MCP 接続設定ではない。ここへ書いても【Code】タブには読まれず、silent failure（§10-1）を生むだけ。
- **棄却根拠2（構造）**：Desktop は `…/AppData/Roaming/Claude/claude-code/<ver>/claude.exe` を同梱し、【Code】タブはこの **Claude Code 本体**を Windows home で実行する。Claude Code の user スコープ MCP は `$HOME/.claude.json` のトップレベル `mcpServers`（`claude mcp add --scope user --transport http` の書込先を、HOME を差し替えた非侵襲プローブで確認）。ゆえに【Code】タブ Local は (b) Windows Terminal と**同一の** `C:\Users\<user>\.claude.json` を読む。
- これは ADR-0002 が prompt 側で確認した「(c)-Local は home-keyed な Windows グローバル設定を読む」性質の、connection 側への**構造的延長**である。新規の対立決定ではなく、ADR-0002 の射程を connection へ広げて §8-5 の内部不整合を正すもの。

### 残実証（honest scope）

【Code】タブ GUI を実際に開いての最終確認（kage://me が【Code】タブ Local で引けるか）は **Phase 3 の init テスト**で取る。万一反証された場合は本 ADR を改訂し、`desktop` 独立ターゲットを復活させる。現時点の根拠（ADR-0002 の prompt 収束 + 同梱 claude.exe + `.claude.json` 配線形式のプローブ確認 + `claude_desktop_config.json` の mcpServers 非保持）は、実装を進めるに足る強度と判断する。

## Consequences

- §8-5 registry 構造を connection **3→2 ターゲット**に簡素化（`wsl`/`winTerm`/`desktop` → `wsl`/`windows`）。jsonc 例を更新し、connection 配線先＝`.claude.json` トップレベル `mcpServers` を明記。
- §8-5 残課題（connection 所在）を解消。
- §7-4 MSIX 吸収は Desktop **アプリ本体**（§3.2 コンテスト提出版の差別化）に限定される。【Code】タブ MVP の connection は `.claude.json` 経由で **MSIX 非依存**（実機も `AppData/Local/AnthropicClaude` の非 MSIX 標準インストールを確認）。
- 付録「検証戦略」の「(c) Desktop パス確認 / 各面設定ファイルの実パス確認」を解消済みへ移す。
- connection の `key` のみ削除（§8-5）・空 `mcpServers` 残骸許容（§8-5）の方針はそのまま有効。`isCreated` を connection が持たない非対称（§8-5）も不変。

---

> [!info] 正典側の対応
> 基本設計書 §8-5 には結論（connection も (b) に収束・配線先は `.claude.json` トップレベル `mcpServers`）の最小記述のみを残し、棄却した `claude_desktop_config.json` 案の経緯・実機プローブの詳細は本 ADR へ移送する。
