# ADR-0003: kage serve のライフサイクルを foreground に限定（常駐デーモンの棄却）

- **Status**: Accepted (2026-06-29)
- **関連**: 基本設計書 §5-6（結論の所在）／ §5-3（上流決定）／ §7-3（無言失敗の補償）／ §10-1（堅牢性との整合）

## Context

`kage serve` が MCP サーバを起動するとき、そのサーバの生存期間（ライフサイクル）をどう設計するかを決める必要があった。性質の異なる選択肢が3つある。

- **foreground**：serve を実行したターミナルが開いている間だけ生存。
- **常駐デーモン / systemd**：ターミナルを閉じても生存し続ける。
- **client-spawned**：クライアント（Claude Code）がサーバプロセスを spawn・管理する。

このうち **client-spawned は本決定で吟味して退けたのではなく、上流の §5-3 で stdio を不採用とした時点で連動して消滅した**。client-spawned は stdio が前提とするライフサイクルであり、HTTP（localhost）で1インスタンスを OS 間共有する方針を選んだ時点で盤上から外れる。棄却の理由は §5-3 側にあるため、本 ADR では foreclosed（先行決定の帰結として消えた選択肢）として扱い、対等な棄却案には数えない。

よって本決定で実際に比較・棄却の対象となるのは **foreground と 常駐デーモン の2択**である。

## Decision

`kage serve` のライフサイクルを **foreground** に限定する。serve を実行したターミナルが生きている間だけサーバが生存し、常駐デーモン化／systemd 連携は採らない。

## 棄却した代替案：常駐デーモン / systemd 連携

ターミナルを閉じてもサーバが生き続けるため、「起動し忘れて無言で失敗する」事故を構造的に減らせる——堅牢性の観点では魅力的で、実在する利点を持つ案である。これを次の二重根拠で棄却する。

### 根拠1（経営軸）：締切に対してコストが見合わない

常駐デーモン化・systemd 連携は、プロセスの常駐管理・自動起動・ログ管理などの実装を要する。U-22 締切（8/31）の制約下で、この実装コストは MVP の成功の定義への寄与に対して過大である。foreground は最小実装で成立する。

### 根拠2（構造軸）：WSL2-VM 寿命がボトルネックになり、常駐の堅牢性メリットが限定的

常駐デーモンが買おうとする堅牢性は「ターミナルを閉じてもサーバが生き続ける」ことである。しかし WSL2 環境では、その上位に **VM の寿命**という、ユーザーが完全には制御しない境界が存在する。

- WSL2 の VM は稼働プロセスが無くなると自動シャットダウンし、マシン再起動や `wsl --shutdown` でも落ちる。VM が落ちればその中の常駐プロセスも当然落ちる。
- VM 停止をまたいでサーバを生かし続けるには、VM 起動時の自動起動配線（systemd 連携など）まで踏み込む必要があり、これは根拠1のコストをさらに膨らませる。

つまり、素の常駐化が foreground に対して上積みできる堅牢性は「ターミナル束縛の解除」までで、より大きな目標である「**VM ライフサイクルをまたいだ常時稼働**」には届かない。kage が守りたい堅牢性の本丸は VM 寿命に上限を握られており、常駐化はその一部しか買えない。

### 2根拠の関係（なぜ二重で要るか）

- **根拠1単独**なら「締切後に常駐化すればよい」で覆る——コストは時間で解ける性質の制約だから。
- **根拠2がこれを塞ぐ**：時間をかけても、素の常駐が買える堅牢性は VM 寿命で頭打ちになり、頭打ちを越えるにはさらにコスト（VM 起動時自動起動）が要る。利得そのものが構造的に限定される。
- コスト（時間で解ける）と構造（環境が握る上限）の二層で退けているため、この棄却は**締切依存の暫定判断ではなく、この環境における恒久的な判断**として読める。

## Consequences

- foreground は「**WSL が稼働している前提**」を制約として受け入れる。VM シャットダウン後のサーバ復帰は、ユーザーの次回 `kage serve` に委ねる（自動復帰は持たない）。
- foreground は構造的弱点として「**起動忘れによる無言の失敗**」（serve し忘れ → resource を引けない → Claude Code がコンテキスト無しで応答）を招く。これは §7-3 の三状態明示トリガーで補償し、§10-1「silent failure を避ける」と整合させる。**棄却で生じた弱点を別機構で埋める**という対の設計になっている点が要。
- client-spawned を foreclosed として §5-3 に紐付けたことで、「なぜ kage は Claude Code にサーバ起動を任せないのか」という問いは、ライフサイクルではなく**トランスポート決定（§5-3）の問題**として読めるようになる。

---

> **正典側の対応**：基本設計書 §5-6 には foreground の結論・採用根拠（実装最小／プロセスの生死が可視で §10-1 と整合）・制約（WSL 稼働依存）・弱点と §7-3 補償の最小記述のみを残し、daemon 棄却の二重根拠・WSL2-VM 寿命の考察・client-spawned の消滅経緯は本 ADR が保持する。
