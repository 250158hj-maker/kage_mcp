# CLAUDE.md — kage 開発規律

> このファイルは kage リポジトリで Claude Code が従う**開発規律**である。 設計の「何を・なぜ」は正典（`docs/基本設計書.md`）が持つ。 本書はそれを**参照で指すだけ**で、内容を再説明しない（再説明は SSoT 違反）。

---

## 1. ドキュメント体系と正本の優先順位

|ドキュメント|場所|持つもの|
|---|---|---|
|正典（基本設計書）|`docs/基本設計書.md`|何を作るか・なぜそう設計したか。**最終権威**|
|ADR|`docs/references/ADR-000N.md`|何を**却下**したか。正典を補強する|
|Arcana.md|リポジトリ外。ユーザーメモリ層（`~/.claude/CLAUDE.md`）から `@import` で供給。**本リポジトリには置かない／commit しない**|汎用原則（再利用可能な設計判断の軸）|

**優先順位：正典 > ADR > Arcana.md。**

規律：

- 設計判断は**まず正典を引く**。
- 正典に書かれていなければ、勝手に確定させない。§3 の「自走の境界」に従い、止めて表面化する。
- 正典の節番号（§N）と ADR は対応している。本書が `§N` を引いたら、根拠はそこにある。重複させない。

## 2. 不変条件（コードが絶対に破ってはならない）

> 全Agentic自走では人手のレビュー網が無い。以下はガードレールであり、 **抵触する実装・リファクタは提案も含めて行わない。** 抵触の可能性が見えたら止めて報告。

1. **SSoT** — `ABOUTME.md` が唯一の源。`registry.json` は**再生成可能なキャッシュ**であって真実ではない。冪等性判定で registry を信頼の根拠にせず、**現実のファイルを正**とする（§8-4 / §9-2）。
2. **pull, not push** — コンテキスト実体を複製しない。`CLAUDE.md` へ書き込むのは取得を促す**ポインタのみ**（§7-3 / §12）。
3. **silent failure 禁止** — 失敗を無言にしない。`self-heal`（黙って直す）は **①終状態が一意 ∧ ②他者領域に非侵襲** の**両立時のみ**。一方でも欠ければ `stop` して観測者（ユーザー / AI エージェント）へ明示する（§10-1）。
4. **自分の footprint しか触らない** — connection 側は `mcpServers.kage` キーのみ操作し、他サーバ設定に触れない。`.claude/` ディレクトリは削除しない（§8-5 / §9-3）。
5. **固定ポート衝突は逃げない** — 別ポートへフォールバックせず、占有プロセスの調査・停止を案内して終了する（§5-7 / 動的ポート棄却の導出は ADR）。

## 3. 今回の開発モード：全 Agentic 自走

本フェーズの目的は**完成**。実装・スパイク・配管・リファクタは**承認待ちせず自走してよい**。

ただし**自走してよい範囲は §10-1 と同型**である：

||進む（self-heal 相当）|止めて報告（stop 相当）|
|---|---|---|
|条件|取るべき手が**一意**で、不変条件・他者領域に**非侵襲**|一意でない（正典が答えを持たない設計判断）／不変条件に触れる／他者の設定領域に触れる、のいずれか|

→ **「承認不要で進む」と「黙って重大判断を確定させる」は別物。** 後者は禁止。 → これは HITL の復活ではなく、kage の設計哲学（§10-1）を開発プロセス自身に適用したもの。

## 4. 設計書の残り3観点（UX / セキュリティ / 検証戦略）の詰め方

これらは正典で**未確定**（付録）。次の順で進める：

1. Agentic スパイク（**使い捨てコード**）で具体的な問いを炙り出す。
2. 出た問いを**正典 / ADR で解決**する。
3. 解決した核を実装する。

**出力の畳み先（本書には書かない）：**

- 結論・判断基準 → **正典**に追記。
- 却下した代替案の理由（「Y ゆえ X を却下」の形になるもの）→ **新規 ADR**（ADR-0006 以降）。却下理由を含まないものは ADR ではなく設計メモ。
- 汎用原則 → **Arcana.md**。
- 詳細設計は**正典化しない**。使い捨ての足場として書き、durable な残骸だけ ADR / インラインコメントへ残す。

## 5. 技術スタックの制約

- 言語：**Rust**（§5-1）。
- MCP 層：**rmcp（公式 SDK）**。MCP プロトコル実装を**自作しない**（§5-2）。
- 配信：**HTTP streamable / localhost / 固定ポート / foreground serve**（§5-3, §5-6, §5-7）。
- コンテキスト形式：**純 Markdown、無加工配信**（§5-4, §8-1）。

---

## 6. コーディング規約

- **コミットメッセージ：日本語**（変更の意図を自分で追えることを優先）。
- **コミット粒度：1 つの関心事で 1 コミット。** 「全部まとめて」を避ける。
- **コメントは「なぜ」を書く。** 不変条件（§2）に関わる箇所は、該当する正典の節番号 / ADR を添える。

---

## 7. タスク完了の作法

- 意味のある単位を終えたら**止まって報告する**（自走 ≠ 無停止。報告は承認待ちではなく可視化のため）。
- 報告に含める：**やったこと / 触れた不変条件があれば該当節 / 次の自走候補**。
- 不明点は黙って埋めず明示し、制約内の最善案を**推奨順に複数提示**する。
